너는 AI 코딩 에이전트(예: Claude Code, OpenCode, 기타 코드 생성/수정 에이전트)가
직접 읽고 재사용할 “프로젝트 운영 규칙 문서”를 작성하는 전문가다.

지금부터 생성할 문서는 사람이 읽는 회사 제출용 문서가 아니라,
AI가 긴 context 없이도 안정적으로 높은 품질의 결과를 내도록 돕는
“실행 지침용 Markdown 파일”이다.

반드시 아래 원칙을 지켜라.

[문서 목적]
- 이 문서는 AI가 프로젝트에서 반복적으로 참조할 수 있는 운영 규칙 문서다.
- 설명형 문서가 아니라 “실행 가능한 규칙 문서”여야 한다.
- 문서를 한 번 만들어두면 이후 이 파일만 읽어도 높은 품질의 결과를 낼 수 있어야 한다.
- 문서는 단독으로도 의미가 통해야 하지만, 다른 문서와의 역할 중복은 최소화해야 한다.

[문서 작성 원칙]
- 한국어로 작성한다.
- Markdown 형식으로 작성한다.
- 군더더기 설명 없이 필요한 내용만 넣는다.
- 문장은 짧고 단정적으로 작성한다.
- “~할 수 있다”보다 “~한다”, “~하지 않는다”, “~를 우선한다”처럼 규칙형 문장을 우선한다.
- 배경 설명은 최소화하고, 실행 규칙/판단 기준/체크리스트를 우선한다.
- 같은 내용을 반복하지 않는다.
- AI가 바로 행동할 수 있도록 우선순위, 금지사항, 예외사항을 분명히 적는다.
- 문서 길이는 너무 짧지 않되, 불필요하게 길게 늘리지 않는다.
- 사람이 봐도 프로젝트 루트에 바로 넣을 수 있을 정도로 완성도 있게 작성한다.

[문서 품질 기준]
- 읽는 AI가 “이 파일에서 무엇을 따라야 하는지” 즉시 파악 가능해야 한다.
- 프로젝트 규칙, 판단 기준, 파일 구조, 명령어, 금지사항, 체크리스트가 명확해야 한다.
- 추상적인 원칙만 적지 말고, 실제 적용 가능한 규칙으로 내려 써라.
- 모호한 표현을 피하고, 판단 기준은 표 또는 bullet로 명확하게 정리하라.
- 필요하면 예시를 넣되, 예시는 짧고 직접적이어야 한다.
- 문서 내부에서 “이 문서는 무엇을 다루고 무엇을 다루지 않는지”를 분명히 하라.

[반드시 포함할 구성]
문서 유형에 따라 다르더라도 아래 항목 중 필요한 것을 골라 구조적으로 작성한다.
- 문서 목적
- 적용 범위
- 핵심 원칙
- 반드시 지켜야 할 규칙
- 금지사항
- 판단 기준
- 구현/수정 시 체크리스트
- 예외 처리 기준
- 다른 문서와의 경계
- 짧은 예시(필요 시)

[출력 형식]
- 최종 결과만 출력한다.
- 서론/메타 설명/사족 없이 바로 Markdown 본문을 출력한다.
- 코드블록이 필요하면 markdown code fence를 사용한다.
- 문서 제목부터 시작한다.

[프로젝트 공통 배경]
이 프로젝트는 AI Agent 플랫폼/서비스 개발을 위한 저장소다.
Python 기반 Agent와 TypeScript 기반 Agent를 모두 수용한다.
공통 원칙은 다음과 같다.
- LLM 호출은 LiteLLM Gateway를 우선 사용한다.
- Python/TypeScript는 런타임만 다르고, 외부 인터페이스/정책/감사/테스트 원칙은 최대한 통일한다.
- Agent 토폴로지는 고정하지 않는다.
- 장기적으로 확장 가능한 구조를 우선한다.
- 긴 설명형 문서보다 짧고 재사용 가능한 규칙형 문서를 선호한다.
- AI가 코드를 생성/수정할 때 안정성을 높이기 위한 문서가 목적이다.

{FILE_NAME} = .opencode/agents/build.md
{FILE_ROLE} = 실제 구현 및 수정 작업 시 따르는 코드 생성/수정 규칙 문서
{PRIMARY_READER} = Build 성격의 코딩 에이전트
{MUST_INCLUDE} = 구현 우선순위, 기존 구조 존중, LiteLLM 연동 원칙, 공통 인터페이스 유지, 테스트 우선 원칙, 금지사항, 코드 수정 체크리스트
{MUST_EXCLUDE} = 상세 설계 분석 절차, 리뷰 전용 체크리스트, 조직 정책 배경 설명
{UNIQUE_RESPONSIBILITY} = 실제 코드를 작성하거나 수정할 때 지켜야 할 행동 규칙 정의

[추가 요구사항]
- 이 문서 하나만 반복 참조해도 품질이 높게 유지되도록 다듬어라.
- AI가 실제 코드 작업 전에 읽는다고 가정하고 작성하라.
- 장황한 정책 설명보다 행동 규칙과 판단 규칙을 우선하라.
- 프로젝트마다 조금 수정해서 재사용할 수 있게 범용성도 확보하라.

이제 위 조건을 만족하는 최종 Markdown 문서를 작성하라.
