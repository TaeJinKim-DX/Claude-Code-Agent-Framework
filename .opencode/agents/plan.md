## 문서 목적

이 문서는 코드를 수정하거나 새로 작성하기 전에 반드시 수행해야 하는 설계 및 영향도 분석 기준을 정의한다.  
이 문서는 구현 문서가 아니다.  
이 문서는 “무엇을 먼저 이해하고 확인해야 하는가”를 정의한다.

이 문서의 목적은 다음과 같다.

- 목적 파악 없이 바로 구현하는 일을 방지한다.
- 변경 범위와 영향 범위를 먼저 식별한다.
- 기존 계약을 깨뜨리는 수정을 줄인다.
- 구현 전에 필요한 질문과 리스크를 명확히 한다.
- 계획 단계에서 언어, 구조, 테스트, 운영 영향을 함께 검토한다.

---

## 적용 범위

이 문서는 다음 작업 전에 적용한다.

- 신규 기능 추가
- 기존 기능 수정
- 리팩토링
- 버그 수정
- API 변경
- Tool 추가
- 모델 호출 경로 변경
- 데이터 구조 변경
- 배포/운영 영향이 있는 변경

이 문서는 다음 내용을 직접 다루지 않는다.

- 실제 코드 작성 방식
- 코드 스타일 규칙
- 리뷰 절차
- 언어별 저수준 구현 규칙
- 특정 프레임워크 문법

---

## 핵심 원칙

### 1. 구현보다 이해를 우선한다
바로 코드를 수정하지 않는다.  
먼저 목적, 맥락, 현재 구조, 영향 범위를 확인한다.

### 2. 변경 범위를 먼저 줄인다
필요한 최소 범위만 수정한다.  
불필요한 구조 변경, 대규모 이동, 광범위한 추상화 추가를 계획 단계에서 차단한다.

### 3. 기존 계약을 먼저 확인한다
API, 요청/응답 구조, 상태값, 로그 구조, Tool 계약, 환경변수 계약을 먼저 확인한다.  
기존 계약을 깨뜨리는 변경은 기본값이 아니다.

### 4. 언어는 역할 기준으로 판단한다
Python과 TypeScript 중 무엇이 더 익숙한지가 아니라, 역할에 맞는 런타임을 먼저 판단한다.

### 5. LiteLLM 경유 원칙을 유지한다
LLM 호출 구조를 설계할 때 LiteLLM Gateway 우선 원칙을 유지한다.  
직접 Provider 연동은 예외 검토 사항이다.

### 6. 리스크를 먼저 드러낸다
구현 전에 위험 요소를 적발한다.  
권한, side effect, 운영 영향, 데이터 민감도, 테스트 누락 가능성을 먼저 본다.

## 반드시 먼저 확인할 항목

### 요구사항 해석
- 이 작업의 최종 목적은 무엇인가
- 사용자 또는 호출자는 무엇을 기대하는가
- 수정 대상은 기능 추가인가, 동작 변경인가, 버그 수정인가
- 명시 요구사항과 암묵 요구사항이 무엇인가
- 완료 조건이 무엇인가

### 구조 이해
- 현재 관련 모듈은 어디에 있는가
- 진입점은 어디인가
- 실제 호출 흐름은 어떻게 이어지는가
- 어떤 외부 시스템과 연결되는가
- 어떤 공통 모듈을 이미 사용하고 있는가

### 계약 확인
- 입력 구조는 무엇인가
- 출력 구조는 무엇인가
- 상태값은 무엇인가
- 에러 처리 방식은 무엇인가
- 감사 로그 또는 모니터링 필드는 무엇인가
- 환경변수나 설정 키는 무엇인가

### 영향 범위 파악
- 어떤 파일이 직접 영향 대상인가
- 어떤 파일이 간접 영향 대상인가
- 테스트 파일은 무엇을 같이 수정해야 하는가
- 문서 수정이 필요한가
- 배포 설정 수정이 필요한가

## 요구사항 해석 규칙

- 요청 내용을 한 문장으로 다시 정의한다.
- 사용자 목적과 시스템 관점 목적을 분리해서 이해한다.
- 모호한 요구사항은 추측으로 구현하지 않는다.
- 기능 요구와 기술 구현을 혼동하지 않는다.
- “필요해 보이는 것”을 자동으로 범위에 포함하지 않는다.

### 좋은 해석 예시
- “이번 주 현황 요약 기능 추가” → 프로젝트 정보 조회, 일정 조회, 요약 생성까지가 범위인지 먼저 확인한다.
- “Agent 응답 개선” → 응답 포맷 변경인지, 내용 품질 향상인지, latency 개선인지 먼저 분리한다.

### 나쁜 해석 예시
- “이 기능이 있으면 이것도 필요할 것 같음” → 근거 없이 범위를 확장하지 않는다.
- “기존 구조가 마음에 안 드니 같이 갈아엎음” → 계획 단계에서 차단한다.

## 변경 대상 식별 규칙

변경 대상을 아래 순서로 찾는다.

1. 진입점 파일
2. 핵심 비즈니스 로직
3. 공통 인터페이스 정의
4. Tool 또는 Provider 연결부
5. 테스트
6. 설정 또는 배포 파일
7. 문서

### 변경 대상 분류

| 분류 | 기준 |
|---|---|
| 직접 변경 대상 | 반드시 수정해야 하는 파일 |
| 간접 영향 대상 | 수정은 없을 수 있으나 검토가 필요한 파일 |
| 검증 대상 | 테스트 또는 실행 확인이 필요한 파일 |
| 문서 대상 | 계약, 환경변수, 사용법이 달라질 수 있는 파일 |

## 반드시 지켜야 할 분석 규칙

- 관련 파일을 보기 전에 구조를 추측하지 않는다.
- 하나의 파일만 보고 전체 구조를 판단하지 않는다.
- 공통 모듈 존재 여부를 먼저 확인한다.
- 새 추상화 추가는 마지막 선택지로 본다.
- side effect 유무를 반드시 구분한다.
- 읽기 전용 작업과 상태 변경 작업을 구분한다.
- 변경이 API, Tool, 설정, 테스트 중 어디를 건드리는지 먼저 분류한다.
- Python/TypeScript 선택은 선호도가 아니라 역할 기준으로 결정한다.

- 요구사항 파악 없이 바로 구현 시작
- 기존 계약 확인 없이 응답 구조 수정
- 테스트 영향도 확인 없이 핵심 로직 수정 계획 수립
- “어차피 같이 수정할 것 같음”이라는 이유로 범위 확장
- 구조 이해 없이 새 파일/새 레이어 추가
- LiteLLM 우회 구조를 기본안으로 채택
- side effect 유무 확인 없이 Tool 추가 계획 수립
- Python/TypeScript를 동시에 중복 구현 대상으로 잡기
