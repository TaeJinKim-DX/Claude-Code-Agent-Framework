## 문서 목적

이 문서는 실제 코드 작성 및 수정 작업 시 따라야 하는 구현 규칙을 정의한다.  
이 문서는 설계 분석 문서가 아니다.  
이 문서는 리뷰 문서가 아니다.  
이 문서는 “어떻게 안전하게 구현하고 수정할 것인가”를 정의한다.

이 문서의 목적은 다음과 같다.

- 구현 단계의 행동 기준을 통일한다.
- 기존 구조와 계약을 보존한다.
- 불필요한 변경과 과도한 추상화를 줄인다.
- LiteLLM, 공통 인터페이스, 테스트 원칙을 일관되게 유지한다.
- 반복적으로 읽어도 안정적인 코드 수정 품질을 유지한다.

---

## 적용 범위

이 문서는 다음 작업에 적용한다.

- 신규 코드 작성
- 기존 코드 수정
- 버그 수정
- 기능 확장
- 리팩토링
- Tool 추가
- Provider 연결 수정
- API 핸들러 수정
- 테스트 코드 추가 및 수정

이 문서는 다음 내용을 직접 다루지 않는다.

- 요구사항 분석 절차
- 영향도 분석 절차
- 리뷰 전용 상세 점검 기준
- 조직 정책 배경 설명
- 언어별 저수준 문법 설명

## 핵심 원칙

### 1. 기존 구조를 우선한다
기존 디렉터리 구조, 책임 분리, 공통 모듈을 먼저 따른다.  
새 레이어, 새 추상화, 새 패턴은 마지막 선택지로 본다.

### 2. 최소 변경 원칙을 따른다
필요한 범위만 수정한다.  
작업 목적과 무관한 리팩토링을 함께 수행하지 않는다.

### 3. 공통 계약을 유지한다
요청/응답 구조, 상태값, 에러 구조, 로그 구조는 공통 기준을 유지한다.  
기존 호출자에 영향을 주는 변경은 기본값이 아니다.

### 4. LiteLLM Gateway를 우선한다
LLM 호출은 LiteLLM Gateway 경유를 기본으로 구현한다.  
직접 Provider SDK 연동은 예외 상황이 아니면 추가하지 않는다.

### 5. 테스트를 구현의 일부로 본다
핵심 로직 수정 시 테스트도 함께 수정한다.  
테스트 없이 핵심 동작을 바꾸지 않는다.

### 6. side effect를 명확히 구분한다
읽기 전용 처리와 상태 변경 처리를 섞지 않는다.  
상태 변경, 외부 호출, 파일 수정, 알림 발송은 명시적으로 드러나야 한다.

### 7. AI 친화적 구조를 유지한다
구현은 사람이 읽기 쉬워야 하고, 다음 AI가 이어서 수정하기 쉬워야 한다.  
숨은 규칙, 암묵적 동작, 과도한 분기를 줄인다.

## 구현 우선순위

코드 작업은 아래 우선순위를 따른다.

1. 기존 계약 보존
2. 정확한 동작 구현
3. 테스트 보장
4. 로그 및 에러 처리 유지
5. 구조 단순성 유지
6. 성능 최적화
7. 새로운 추상화 추가

### 우선순위 해석
- 성능보다 계약 안정성을 우선한다.
- 새로운 패턴 도입보다 기존 구조 일관성을 우선한다.
- “더 예쁘게”보다 “덜 위험하게”를 우선한다.

## 반드시 지켜야 할 구현 규칙

### 공통 규칙
- 변경 전에 관련 파일과 호출 흐름을 확인한다.
- 기존 공통 모듈이 있으면 재사용한다.
- 동일 역할의 로직을 새로 복제하지 않는다.
- 입력 검증 없이 비즈니스 로직이나 Tool 호출로 진입하지 않는다.
- 민감정보를 로그에 기록하지 않는다.
- 설정값, 키, URL을 코드에 하드코딩하지 않는다.
- 예외는 조용히 삼키지 않는다.
- 실패 경로도 명시적으로 처리한다.
- side effect가 있는 함수는 이름과 구조에서 드러나게 작성한다.
- 테스트가 있는 구조를 무시하고 임시 우회 코드를 남기지 않는다.

### 인터페이스 규칙
- 공통 요청/응답 구조를 유지한다.
- 상태값은 기존 표준을 따른다.
- 에러 응답 구조를 임의로 바꾸지 않는다.
- Tool 입력/출력은 가능하면 구조화된 형태로 유지한다.
- 문자열 파싱 의존 구현은 최소화한다.

### LiteLLM 규칙
- 모델 호출은 LiteLLM Gateway 경유를 기본으로 한다.
- 모델 식별자는 Gateway alias 사용을 우선한다.
- Provider 직접 호출은 예외 사유가 명확할 때만 추가한다.
- LiteLLM 관련 환경변수 키 이름은 임의로 바꾸지 않는다.
- Gateway 우회 구현을 넣을 경우 반드시 코드나 문서에 예외 사유를 남긴다.

### 테스트 규칙
- 핵심 로직 변경 시 테스트를 같이 수정한다.
- 새 기능에는 최소 1개 이상의 정상 경로 테스트를 추가한다.
- 에러 경로가 중요하면 실패 케이스 테스트도 추가한다.
- mock 또는 offline 경로가 존재하면 그것도 유지한다.
- 테스트가 깨지면 원인을 설명하지 않은 채 무시하지 않는다.

## 구현 시 판단 기준

### 새 파일을 만들지 말고 기존 파일을 수정해야 하는 경우
- 기존 파일 책임 범위 안에 있는 변경이다.
- 관련 로직이 이미 같은 모듈에 모여 있다.
- 새 파일을 만들면 오히려 탐색 비용이 커진다.

### 새 파일을 만들어야 하는 경우
- 기존 파일 책임 범위를 명확히 넘는다.
- 공통 재사용 가능성이 높다.
- 기존 파일이 과도하게 커져 응집도가 깨진다.

### 공통 모듈로 올려야 하는 경우
- 둘 이상 위치에서 같은 로직이 반복된다.
- 언어나 기능이 달라도 의미상 같은 계약을 갖는다.
- 한 번 수정할 때 여러 곳을 동시에 고쳐야 하는 구조가 된다.

### 공통 모듈로 올리면 안 되는 경우
- 지금은 한 곳에서만 사용한다.
- 재사용 가능성이 추측 수준이다.
- 추상화 비용이 실제 이득보다 크다.

## 코드 수정 체크리스트

### 수정 전
- 변경 목적을 한 문장으로 설명할 수 있는가
- 직접 수정할 파일을 식별했는가
- 간접 영향 파일을 식별했는가
- 기존 인터페이스를 확인했는가
- 테스트 영향 범위를 확인했는가

### 수정 중
- 범위를 불필요하게 넓히지 않았는가
- 기존 공통 모듈을 우선 사용했는가
- LiteLLM 원칙을 유지했는가
- side effect 유무가 코드에서 드러나는가
- 예외 처리와 로그 처리가 유지되는가

### 수정 후
- 정상 경로 테스트를 확인했는가
- 실패 경로도 최소한 검토했는가
- 응답 구조나 상태값이 깨지지 않았는가
- 민감정보 노출 가능성이 없는가
- 문서나 설정 수정이 필요하면 반영했는가

## 금지사항

- 목적과 무관한 리팩토링 동시 수행
- 기존 인터페이스를 조용히 변경
- 테스트 없는 핵심 로직 수정
- LiteLLM 대신 직접 Provider 호출을 기본 구현으로 채택
- 환경변수 없이 코드 내부 상수로 연결 정보 작성
- 공통 모듈이 있는데 유사 로직을 새로 복제
- TODO만 남기고 핵심 실패 경로를 미구현 상태로 종료
- 임시 디버그 코드, 임시 프린트, 임시 우회 로직을 정리 없이 남김
- 긴 설명형 주석으로 동작을 대신 설명
- 읽기 전용 로직과 상태 변경 로직을 한 함수에 무분별하게 혼합

## 예외 처리 기준

### 1. 긴급 수정
빠른 수정이 필요할 수 있다.  
단, 아래는 생략하지 않는다.

- 직접 영향 파일 확인
- 최소 수정 범위 유지
- 기본 테스트 확인
- 임시 처리라면 명시적 표기

### 2. LiteLLM 우회가 필요한 경우
아래 조건을 모두 만족할 때만 허용한다.

- 기술적으로 Gateway 경유 구현이 현재 불가능하다.
- 우회 사유가 분명하다.
- 추후 정리 방향을 코드 또는 문서에 남긴다.

### 3. 테스트를 당장 추가하기 어려운 경우
아래를 남긴다.

- 왜 지금 테스트 추가가 어려운지
- 어떤 리스크가 있는지
- 이후 어떤 테스트를 추가해야 하는지

단, 이 예외는 기본값이 아니다.

## 짧은 구현 예시

### 좋은 예시
- 기존 Provider 래퍼가 있으면 그 안에서 LiteLLM 호출만 확장한다.
- 응답 구조를 유지한 채 result 내부 필드만 안전하게 확장한다.
- 테스트 파일도 함께 수정해 새 동작을 반영한다.

### 나쁜 예시
- 같은 역할의 새로운 Provider 클래스를 임의 이름으로 하나 더 만든다.
- 직접 OpenAI SDK 호출을 추가하고 LiteLLM 경로는 그대로 둔다.
- 기능 하나 바꾸면서 unrelated 리팩토링을 같이 수행한다.

기존 구조를 먼저 따른다.  
최소 범위만 수정한다.  
공통 계약을 유지한다.  
LiteLLM Gateway를 우선한다.  
테스트를 같이 수정한다.  
불필요한 추상화는 만들지 않는다.  
민감정보를 남기지 않는다.  
side effect는 명확히 드러내고 제어한다.
